# 消息终端-产品计划书

#### 桌面消息终端（Desktop Notification Terminal）



## 一、项目概述

### 1.1 项目目标

开发一款基于 **ESP32-S3** 的桌面消息终端设备，用于接收来自手机 App、USB、本地 HTTP 接口的消息，并通过不同显示介质进行提醒与展示。

第一阶段重点实现：

- 手机端像**智能手环**一样接收 Android 系统通知
- 用户可在手机 App 中选择哪些应用允许推送
- 手机 App 通过 **BLE** 将通知转发到设备
- 设备进行灯光提醒或屏幕显示
- 同时支持 **USB** 和 **HTTP** 作为额外消息输入方式

### 1.2 产品定位

这不是手环，也不是纯开发板工具，而是一个：

**“桌面型多通道消息接收与展示终端”**

它的核心价值在于：

- 将手机重要通知转移到桌面设备上
- 降低查看手机频率
- 以更直观的灯光 / 屏幕方式提示消息
- 支持后续扩展为办公桌面提醒器、专注提醒器、消息公告牌、轻量控制终端

------

## 二、产品愿景

### 2.1 短期目标

做出可用原型，完成如下闭环：

**Android 应用通知 → 手机 App 筛选 → BLE 推送 → ESP32-S3 设备接收 → LED / 屏幕显示**

### 2.2 中期目标

形成可长期使用的桌面设备，支持：

- 多消息队列
- 应用过滤
- 优先级提醒
- 插电小屏展示
- HTTP 局域网推送
- USB 本地配置与调试

### 2.3 长期目标

扩展为一个平台型产品，支持：

- 墨水屏版本
- 电池版便携设备
- 企业内部通知设备
- 桌面状态看板
- 与 PC / Home Assistant / 自动化系统联动

------

## 三、用户场景

### 3.1 核心场景

用户在 Android 手机上安装配套 App，授权通知访问权限，并选择允许推送的应用，例如：

- 微信
- 企业微信
- 钉钉
- Outlook
- Telegram
- 自定义业务 App

当这些应用产生新通知时，手机 App 自动将通知通过 BLE 转发到桌面设备，设备立即以灯光或屏幕显示提醒用户。

### 3.2 扩展场景

- 电脑或服务端通过 HTTP 向设备发送消息
- 本地通过 USB 向设备发送测试消息
- 设备显示状态、待办、提醒、简短消息
- 在工作时段进行重点应用通知提示
- 在静音时段仅记录消息不主动提醒

## 八、核心功能设计

## 8.1 手机通知转发

### 功能描述

手机 App 获取系统通知访问权限后，监听通知事件，仅将用户允许的应用通知转发到设备。

### 能力点

- 应用白名单
- 总开关
- 标题与摘要提取
- 去重
- 长文本裁剪
- 推送失败重试

------

## 8.2 BLE 通信

### 角色

- 手机：GATT Client
- 设备：GATT Server

### 第一版目标

- 手机向设备写入一条通知消息
- 设备返回 ACK
- 支持简单控制指令

### 特征建议

- `NotifyRx`：写入通知
- `AckTx`：返回接收状态
- `Control`：控制命令
- `Status`：设备状态

------

## 8.3 USB 推送

### 目标

作为本地调试与备用输入方式。

### 第一版建议

- 使用 **USB CDC**
- 文本协议或简化 JSON 协议
- 支持发送测试消息、查询状态、清空消息

------

## 8.4 HTTP 推送

### 目标

支持局域网内的消息输入。

### 第一版接口

- `POST /api/message`
- `GET /api/status`
- `POST /api/command`

### 作用

- PC 工具联调
- 自动化系统推送
- 局域网服务集成

------

## 8.5 显示策略

### LED 阶段

- 颜色表示应用类型或消息级别
- 闪烁表示紧急程度
- 支持常亮、呼吸、闪烁

### 小屏阶段

- 显示应用名
- 显示标题
- 显示摘要
- 列表视图

- 状态页

### 墨水屏阶段

- 低频刷新
- 卡片式消息
- 静态状态展示

## 十、统一消息模型

设备内所有输入都转换为统一结构。

```
typedef enum {
    MSG_SRC_BLE,
    MSG_SRC_USB,
    MSG_SRC_HTTP,
} msg_source_t;

typedef enum {
    MSG_TYPE_TEXT,
    MSG_TYPE_ALERT,
    MSG_TYPE_STATUS,
    MSG_TYPE_COMMAND,
} msg_type_t;

typedef struct {
    msg_source_t source;
    msg_type_t type;
    uint32_t id;
    uint32_t timestamp;
    uint8_t priority;
    char app_name[32];
    char title[64];
    char content[256];
} app_message_t;
```

这个结构保证后续不管消息来自哪里，显示与处理方式都一致。

## 十二、开发阶段规划

## 12.1 第一阶段：最小可用原型

### 目标

证明核心链路成立。

### 内容

- ESP32-S3 跑 BLE GATT Server
- Android App 监听通知
- 用户选择应用白名单
- 手机 BLE 写入消息
- 设备 LED 提醒
- 设备 ACK 回传

### 验收标准

- 安装 App 后可选择应用
- 至少 3 个应用通知可稳定转发
- BLE 链路连续运行稳定
- 设备可区分不同消息优先级

- HTTP 接口
- USB 调试接口
- 配置持久化

### 验收标准

- 支持 BLE / USB / HTTP 三通道消息接入
- 屏幕可展示最近消息
- 设备断电重启后配置保留
- 本地调试流程清晰可复现

## 十四、风险与对策

## 14.1 Android 后台限制

### 风险

后台蓝牙连接和服务容易受系统限制。

### 对策

- 优先做前台服务方案
- 结合通知监听与 BLE 稳定连接策略
- 首版只考虑主流 Android 机型

------

## 14.2 BLE 文本传输不稳定

### 风险

通知内容长度不固定，可能导致分包问题。

### 对策

- 第一版限制标题与正文长度
- 自定义简单分包协议
- ACK 与重试机制

------

## 14.3 显示形态变化带来重构

### 风险

从 LED 到小屏再到墨水屏，容易导致代码重写。

### 对策

- 从第一天开始抽象 display 接口
- 禁止在 BLE/HTTP/USB 层直接操作显示

------

## 14.4 产品范围失控

### 风险

一开始想做太多，节奏被拖慢。

### 对策

- 第一版只做通知转发与展示
- 不做云端
- 不做 iOS
- 不做复杂 UI

------

## 十五、成功标准

当满足以下条件时，可以认为项目第一阶段成功：

- 用户安装 Android App 后，可选择指定应用进行推送
- 手机通知可稳定经 BLE 到达设备
- 设备可通过 LED 或屏幕完成可理解的提醒
- USB / HTTP 至少有一个备用输入方式可用
- 设备连续运行稳定，无明显死机或断链问题
- 代码结构已支持未来替换显示设备

## 十八、最终建议

这个项目最合理的起点不是“做一个完美设备”，而是：

**先做出一个像手环那样转发通知、但放在桌面的消息终端。**

也就是：

- 手机 App 只做通知筛选与转发
- ESP32-S3 只做消息接收与展示
- 显示先从 LED 做起
- 小屏是主产品方向
- 墨水屏作为后续分支

这样你能最快启动，而且整个产品路线是稳的。
